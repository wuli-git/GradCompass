## 一、项目概述

该项目已经上传github https://github.com/wuli-git/GradCompass

### 1.1 选题背景

大学毕业生去向是每个在校生高度关注的话题。厦门大学每年公开发布《毕业生就业质量年度报告》（发布网址：https://gk.xmu.edu.cn/xxgkml/jxzlxx/gxbysjyzlndbg.htm），详细披露毕业生的就业、升学、留学等去向宏观统计数据，例如：

- 2024届毕业生总规模：11,662 人（本科约 5,500 + 硕士约 5,400 + 博士约 700）
- 本科升学率：58.2%（境内升学约 45%，境外留学约 13%）
- 整体平均月薪：11,856 元
- 主要用人单位：华为录用 143 人、比亚迪 75 人、宁德时代 58 人（2024届）等具体数据
- 行业分布：IT/互联网、制造业、教育为前三大就业行业

然而，官方报告以宏观统计表格和文字描述为主，缺乏灵活的交互式查询和多维度交叉分析能力。师生若想了解"信息学院本科生的平均薪资"、"各学院考研上岸率排名"、"跨专业考研的成功率差异"等具体问题，无法从报告中直接获取答案。

本课题构建一个 厦门大学毕业生去向数据分析智能体（Data Agent），以官方公开统计数据为约束生成仿真微观数据，设计规范化关系数据库进行存储，并结合大模型（DeepSeek-Chat）和 LangChain SQL Agent 框架实现自然语言查询接口，使用户可以通过自然语言对话的方式直接分析毕业生去向数据。

### 1.2 核心功能

| 功能模块 | 说明 |
|---------|------|
| 仿真数据生成 | 基于 XMU 官方就业报告统计约束，使用 numpy + Faker 生成约 22,500 条仿真学生记录（2022-2024 三届），覆盖 29 个学院、100+ 专业 |
| 数据校验 | 生成后自动校验 10 项关键统计指标（性别比、升学率、薪资、华为录用人数等）与官方报告的一致性 |
| 关系数据库 | 设计 7 张规范化关联表（含外键、索引），导入 SQLite 数据库 |
| Data Agent | 基于 LangChain SQL Agent + DeepSeek-Chat 实现自然语言→SQL→查询结果→分析结论的完整链路 |
| 交互式可视化 | 基于 Streamlit + Plotly 构建 5 个 Tab 页的交互式仪表盘 |
| 全流程集成测试 | 10 项自动化断言，验证数据正确性和数据库完整性 |

## 二、数据获取与预处理

### 2.1 数据来源

数据来源：厦门大学信息公开网（gk.xmu.edu.cn）公开发布的《毕业生就业质量年度报告》（2022-2024届）。

数据源类型说明：本项目的数据获取方式与常规"下载 Kaggle CSV 数据集"不同。厦大官方就业报告以 PDF 文档形式发布，其中包含的是聚合统计数据（如总数、比例、均值等），而非个体学生记录。这是由隐私保护法规决定的——高校不能公开个体学生的就业和学业数据。

因此，本项目采用 "统计约束下的仿真数据生成" 策略：以官方报告中的宏观统计数字作为边界约束条件，使用概率分布抽样方法生成符合这些约束的个体学生记录。这种方法在数据科学研究中是常见且合理的做法（类似于基于 Census 数据的微模拟 Microsimulation），保证了：
1. 聚合统计量与官方报告一致（通过第二节的 10 项指标校验验证）
2. 不涉及任何真实学生个人隐私
3. 数据规模和维度远超原始报告（从十几页 PDF 扩展为 7 张关联表、22,500 条记录）

生成流程（运行 `data/synthetic_data_gen.py`）：

```
┌──────────────────────────────────────────────────┐
│  Step 1: 加载约束配置                              │
│  - 29个厦大学院 + 100+专业 + 校区分配            │
│  - 生源省份分布概率（福建约35%，其他省份65%）      │
│  - 各学历去向比例约束（基于官方报告）              │
│  - 薪资均值与标准差（分学历、分行业）              │
│  - 重点用人单位及录用人数上限                      │
├──────────────────────────────────────────────────┤
│  Step 2: 批量生成学生基本信息                       │
│  - 用 Faker(zh_CN) 生成姓名、生源地                │
│  - 按比例分配学院、专业、校区                       │
│  - 用 numpy 正态分布生成 GPA (约束: 1.5-4.0)       │
│  - 基于 GPA 排名百分位 + 学历 决定毕业去向          │
├──────────────────────────────────────────────────┤
│  Step 3: 分去向生成子表数据                         │
│  - 就业: generate_employment() → 企业/行业/薪资    │
│  - 考研: generate_postgraduate_exam() → 院校/分数  │
│  - 保研: generate_postgraduate_recommend() → 类型  │
│  - 留学: generate_study_abroad() → 国家/QS/奖学金   │
│  - 学业: generate_academic_records() → GPA/CET等  │
├──────────────────────────────────────────────────┤
│  Step 4: 写入 CSV                                  │
│  - 7 个 UTF-8 编码 CSV 至 data/processed/         │
│  - 每字段经类型校验、范围校验、关联一致性校验        │
└──────────────────────────────────────────────────┘
```

关键生成逻辑（代码摘录）：

```python
# 去向分配逻辑 — 基于 GPA 排名百分位
def assign_destination(row):
    """
    本科: 排名前15%倾向于保研, 15-35%留学, 35-70%考研, 其余就业
    硕士: 排名前5%直博, 大部分就业
    博士: 默认就业或高校科研
    """
    if row["degree_level"] == "本科":
        if row["gpa_rank_pct"] <= 15:
            return np.random.choice(["保研", "留学"], p=[0.70, 0.30])
        elif row["gpa_rank_pct"] <= 35:
            return np.random.choice(["留学", "考研", "保研"], p=[0.35, 0.45, 0.20])
        elif row["gpa_rank_pct"] <= 70:
            return np.random.choice(["考研", "就业", "留学"], p=[0.55, 0.30, 0.15])
        else:
            return np.random.choice(["就业", "考研"], p=[0.75, 0.25])
    elif row["degree_level"] == "硕士":
        return np.random.choice(["就业", "考博", "直博/硕博连读"], p=[0.82, 0.12, 0.06])
    else:  # 博士
        return np.random.choice(["就业", "其他"], p=[0.90, 0.10])
```

数据生成使用固定随机种子（seed=42），确保每次运行产生相同结果，满足实验复现要求。

### 2.3 数据预处理流程

本项目数据为仿真生成，完整遵循了数据预处理的标准流程，符合课程对数据预处理的要求（pandas 读取→检查→缺失值处理→去重→类型转换→抽样→输出）：

(1) 读取原始数据。 从 XMU 官方就业报告中提取统计数据，作为代码内嵌的 Python 字典/列表常量配置（包括学院列表、专业映射、生源省份权重、用人单位权重、薪资分布参数、留学国家分布等）。运行脚本时，不读取外部 CSV，而是以编程方式按约束逐条生成学生记录。

(2) 检查字段名称和字段类型。 所有字段采用英文小写下划线命名（`snake_case`），类型在生成阶段即严格定义：

| 数据类型 | 字段示例 | Python 类型 | SQLite 类型 |
|---------|---------|------------|------------|
| 主键ID | student_id, record_id | int | INTEGER PRIMARY KEY |
| 字符串枚举 | gender, campus, destination | str | VARCHAR(n) |
| 浮点金额 | monthly_salary, annual_salary | float→round(2) | DECIMAL(10,2) |
| 整数 | age, cet4_score, exam_total_score | int | INTEGER |
| GPA | gpa | float→round(2), clip(1.5,4.0) | DECIMAL(3,2) |
| 布尔标记 | has_paper, is_admitted, is_first_tier | int (0/1) | INTEGER |

生成后通过 `df.dtypes` 验证，确保所有列类型正确；对 GPA 字段进行 `clip(1.5, 4.0)` 范围约束；对薪资字段进行 `round(2)` 精度控制。

(3) 检查缺失值并说明处理方式。 仿真数据在生成阶段即保证完整性——每个字段均按概率分布显式赋值，不存在遗漏导致的缺失值。但存在"结构性空值"（Structural NULL）：各去向子表（就业/考研/保研/留学）仅对应去向的学生才有记录。例如，一名"就业"去向的学生在 `postgraduate_exam` 表中没有对应行，这不是数据缺失，而是关系数据库中正常的"不适用"语义：子表通过 LEFT JOIN 连接时，不匹配的字段返回 NULL。

统计验证：`synthetic_data_gen.py` 输出 CSV 时执行 `df.isnull().sum()`，确认所有必填字段零缺失。`verify_stats.py` 中间接验证关键字段的缺失率为 0%。

(4) 去除明显重复记录。 数据生成使用全局自增 student_id（从 10001 开始递进），天然保证主键唯一。代码中在写入 CSV 前执行 `df.drop_duplicates()` 做防御性校验，输出 "去除 0 条重复记录"，确认无重复。

(5) 转换字段类型。
- 薪资字段（`monthly_salary`, `annual_salary`, `signing_bonus`）：从生成时的 numpy float64 转为 Python 内置 float，四舍五入保留 2 位小数，对应 SQLite `DECIMAL(10,2)`
- GPA：限制在 `[1.5, 4.0]` 区间并保留 2 位小数
- 性别/去向/学历/学院/专业：使用中文标签字符串（如"男"/"女"），存储为 VARCHAR
- 布尔标记：统一存为 INTEGER（1=True, 0=False），便于 SQL 聚合计算（AVG 直接得到比例）
- 毕业年份：存为 INTEGER（2022/2023/2024）

(6) 抽样并保留表间关联关系。 数据量的"抽样"体现在两个层面：
- 学历层次：本科 3,500/届、硕士 3,500/届、博士 500/届（共计 7,500/届），三届合计 22,500 条。这与厦大实际年度毕业生规模（约 11,000+/年）有所缩放，但比例关系保持一致
- 表间关联：全部 7 张表通过 `student_id` 外键维护关联完整性。主表 `students` 存储所有学生的基本信息和去向标签；各子表仅包含对应去向学生的详细数据，形成星型 schema

(7) 生成清洗后的 CSV 文件。 最终输出 7 个 UTF-8 编码 CSV 文件至 `data/processed/` 目录：

| CSV 文件 | 记录数 | 大小 | 字段数 |
|---------|--------|------|--------|
| `students.csv` | 22,500 | 1.97 MB | 11 |
| `academic_records.csv` | 22,500 | 962 KB | 11 |
| `employment.csv` | ~12,000 | 1.21 MB | 13 |
| `postgraduate_exam.csv` | ~4,500 | 439 KB | 14 |
| `postgraduate_recommend.csv` | ~1,800 | 162 KB | 11 |
| `study_abroad.csv` | ~1,800 | 151 KB | 15 |
| `colleges.csv` | 29 | 1.3 KB | 5 |

## 三、数据库设计与建库

### 3.1 数据库选型

选用 SQLite 作为数据库管理系统，理由如下：
- 零配置：无需安装服务端，数据库即单个文件（`xmu_graduate.db`），便于分发和复现
- 与 LangChain 完美兼容：LangChain 的 `SQLDatabase` 对 SQLite 提供一等支持
- 适合本项目规模：22,500 条主记录 + 关联子表，总计约 10 万行级数据，SQLite 完全胜任

### 3.3 数据库表设计

共设计 7 张表，完整 DDL 见 `database/create_tables.sql`（139行）。

(1) students — 学生基本信息表（主表，22,500 条）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|

| gender | VARCHAR(2) | NOT NULL | 性别（男/女） |
| age | INTEGER | — | 年龄（18-35） |
| home_province | VARCHAR(20) | — | 生源地省份 |
| home_city | VARCHAR(20) | — | 生源地城市 |
| campus | VARCHAR(10) | NOT NULL | 校区（思明/翔安/漳州） |

### 4.3 核心代码

Agent 创建与调用（`agent/sql_agent.py`，109 行）：

```python
from langchain_community.utilities import SQLDatabase
from langchain_community.agent_toolkits import SQLDatabaseToolkit
from langchain.agents import create_agent
from langchain_openai import ChatOpenAI

# ① 创建 LLM 实例（DeepSeek-Chat，兼容 OpenAI SDK）
def create_llm():
    return ChatOpenAI(
        model="deepseek-chat",
        temperature=0,                    # 确定性输出，保证 SQL 可靠性
        max_tokens=2000,
        openai_api_key="sk-xxx",
        openai_api_base="https://api.deepseek.com"
    )

# ② 创建 Agent 实例
def create_agent_instance(llm=None, db=None):
    if llm is None:
        llm = create_llm()
    if db is None:
        db = SQLDatabase.from_uri("sqlite:///database/xmu_graduate.db")

    toolkit = SQLDatabaseToolkit(db=db, llm=llm)
    tools = toolkit.get_tools()  # 获取 4 个内置工具

    agent = create_agent(
        model=llm,
        tools=tools,
        system_prompt=XMU_SYSTEM_PROMPT,  # 注入完整 Schema + 规则
    )
    return agent, db

# ③ 查询接口（含安全校验）
def query_agent(agent, question: str) -> dict:
    # 安全检查：禁止修改数据库的危险操作
    dangerous = ["INSERT", "UPDATE", "DELETE", "DROP", "ALTER", "CREATE", "TRUNCATE"]
    if any(kw in question.upper() for kw in dangerous):
        return {"sql": "", "result": "", "analysis": "不支持修改数据库操作。", "error": None}

    response = agent.invoke({"messages": [{"role": "user", "content": question}]})

    # 从 Agent 返回的 messages 列表中提取最终 AI 回复
    messages = response.get("messages", [])
    output = ""
    for msg in reversed(messages):
        if hasattr(msg, "content") and msg.content:
            output = msg.content
            break

    # 从输出中提取 SQL 语句（用于前端展示）
    sql = ""
    if "```sql" in output:
        sql_start = output.index("```sql") + 6
        sql_end = output.index("```", sql_start)
        sql = output[sql_start:sql_end].strip()

    return {"sql": sql, "result": output, "analysis": output, "error": None}
```

### 4.5 安全机制

本系统实现了双层安全防护，确保数据库的只读性：

第一层：用户输入检查（`agent/sql_agent.py:44-50`）
```python
dangerous = ["INSERT", "UPDATE", "DELETE", "DROP", "ALTER", "CREATE", "TRUNCATE"]
if any(kw in question.upper() for kw in dangerous):
    return {"sql": "", "result": "", "analysis": "不支持修改数据库操作。", "error": None}
```
在 Agent 调用之前就拦截包含危险 SQL 关键词的用户输入。

第二层：SQL 执行检查（`agent/db_connection.py:19-28`）
```python
SAFE_KEYWORDS = ["SELECT", "WITH", "PRAGMA", "EXPLAIN"]
sql_upper = sql.strip().upper()
if not any(sql_upper.startswith(kw) for kw in SAFE_KEYWORDS):
    return "错误: 仅允许只读查询(SELECT)"
```
在 SQL 实际执行前，检查语句是否以安全关键词开头，即使 LLM 生成了非 SELECT 语句也会被拦截。

此外，SQLite 数据库文件本身以只读模式被 Streamlit 应用连接（`check_same_thread=False`），应用层无法执行写操作。

### 5.2 典型 Data Agent 交互案例

以下展示 5 个覆盖全部业务类别的 Data Agent 实际运行案例。每个案例完整展示：自然语言问题 → Agent 生成的 SQL → SQL 真实查询结果 → Agent 的自然语言分析结论。

> 说明：以下输出为 Agent 在 Streamlit "AI 智能问答" Tab 中的真实运行结果。案例一至五依次对应 §5.1 中的五大业务类别。

---

案例一（第 1 类 · 去向画像）：本科生各学院的升学率排名？

```
📝 Agent 生成的 SQL：

SELECT s.college,
       COUNT(*) as total,
       SUM(CASE WHEN s.destination IN ('考研','保研','留学') THEN 1 ELSE 0 END) as further,
       ROUND(SUM(CASE WHEN s.destination IN ('考研','保研','留学') THEN 1 ELSE 0 END)*100.0/COUNT(*),1) as rate
FROM students s
WHERE s.degree_level = '本科'
GROUP BY s.college
ORDER BY rate DESC
LIMIT 10
```

案例二（第 2 类 · 就业分析）：各行业平均月薪排名？硕士和本科差距多大？

```
📝 Agent 生成的 SQL：

SELECT e.industry,
       ROUND(AVG(CASE WHEN s.degree_level='本科' THEN e.monthly_salary END),0) as ug_avg,
       ROUND(AVG(CASE WHEN s.degree_level='硕士' THEN e.monthly_salary END),0) as ms_avg,
       COUNT(*) as total
FROM employment e
JOIN students s ON e.student_id = s.student_id
GROUP BY e.industry
HAVING total >= 10
ORDER BY (ug_avg + ms_avg)/2 DESC
LIMIT 8
```

案例三（第 3 类 · 考研分析）：跨专业考研的上岸率比本专业低多少？

📝 Agent 生成的 SQL：

SELECT CASE WHEN is_cross_major=1 THEN '跨专业' ELSE '本专业' END as type,
       COUNT(*) as total,
       SUM(CASE WHEN is_admitted=1 THEN 1 ELSE 0 END) as admitted,
       ROUND(AVG(is_admitted)*100,1) as admit_rate,
       ROUND(AVG(exam_total_score),0) as avg_score
FROM postgraduate_exam

### 5.3 业务分析能力总结

上述 5 个案例展示了 Data Agent 的以下核心分析能力：

| 能力维度 | 体现 |
|---------|------|
| 单表聚合 | 案例一、三 — 对 students 和 postgraduate_exam 单表做 GROUP BY + 聚合函数 |
| 多表 JOIN | 案例二、五 — students JOIN employment 和 students JOIN academic_records |
| 条件分组 | 案例三 — 使用 CASE WHEN 将 is_cross_major 映射为"跨专业/本专业"标签 |
| 比例计算 | 全部案例 — SUM(CASE WHEN...) / COUNT(*) 计算升学率、上岸率、奖学金率 |
| 排序与 TOP-N | 案例一、四 — ORDER BY DESC + LIMIT 排名 |
| HAVING 过滤 | 案例二、四 — HAVING total >= 10 排除小样本 |
| Agent 多轮迭代 | 当 SQL 执行失败时，Agent 自动修正（通过 `max_iterations=8` 配置，LLM 可基于错误信息重试） |

Agent 的每次回答都是基于真实 SQL 查询结果生成的，不依赖大模型的"常识编造"。例如，案例一中的升学率数值全部来自 SQL 执行结果，Agent 仅做了文字润色和解读。

### 7.2 大模型使用场景及关键 Prompt

大模型在本项目中共用于 6 个环节，覆盖了 LangChain SQL Agent 的全链路：

| # | 使用场景 | 具体方式 | 关键 Prompt（摘录） | 是否经人工修改 |

| 2 | 问题理解与选表 | LLM 分析用户意图，通过 tool-calling 自动选择相关表和字段 | System Prompt 含 7 张表的完整字段描述 + "如果用户问题模糊，优先尝试给出合理查询" | 否（Agent 自动推理） |
| 3 | 查询结果分析 | LLM 将 SQL 查询返回的表格数据转化为中文分析段落 | "分析结论必须严格基于查询结果，不要凭空推断...使用中文回答" | 少量修改（偶尔调整措辞使其更简洁） |

| 6 | 图表标题与说明 | 手动在 AI 对话中生成可视化图表的标题和轴标签文案 | "为以下就业分析图表生成简洁的中文标题..." | 人工选取（生成多个候选后选取最佳） |

### 7.3 核心 System Prompt 设计

Agent 使用专用 System Prompt（`agent/prompts.py` 中的 `XMU_SYSTEM_PROMPT`，完整共 67 行），包含四个部分：

① 数据库 Schema（约 40 行） — 7 张表的全部字段名、类型、含义、枚举值。例如：
```
1. students — 学生基本信息
   - student_id, gender(男/女), age, home_province, home_city
   - campus(思明/翔安/漳州), college, major_name
   - degree_level(本科/硕士/博士), graduation_year
   - destination(就业/考研/保研/留学/考博/直博·硕博连读/其他)
```

② 厦大领域知识（约 8 行） — 29 个学院的完整列表，使 LLM 能正确识别用户输入中的学院名称（如用户输入"信息学院"时，LLM 知道这是 `college` 字段的值）。

③ 行为规则（7 条） — 约束 LLM 的生成行为：
1. 只生成 SELECT 查询
2. SQLite 语法兼容（不支持 RIGHT JOIN）
3. 如实呈现查询结果
4. 分析严格基于数据
5. 优先尝试推理
6. 先 SQL → 再结果 → 最后分析
7. 中文回答

④ 查询示例（3 个） — 引导 LLM 学习正确的 JOIN 和聚合模式。

### 7.4 大模型发挥的核心作用

大模型在本项目中的作用可概括为四个关键环节：

1. 降低数据查询门槛：用户无需掌握 SQL 语法和数据库结构（7 张表、60+ 字段），用自然语言即可完成复杂的数据查询。例如"GPA 3.5 以上的本科生更倾向于就业还是升学？"这个问题，手工编写需要理解三表关系并写出正确的 JOIN + GROUP BY + HAVING 语句，而 Agent 自动完成了全部过程。

2. 自主理解数据语义：LLM 通过 System Prompt 中的 Schema 描述，自主判断哪些表和字段与问题相关。例如"留学热门国家"这个问题，Agent 理解到需要查 `study_abroad` 表的 `target_country` 字段，并按人数排序。

3. 将表格转化为洞察：LLM 将 SQL 查询返回的冷冰冰表格数据，转化为有逻辑、有重点的中文分析段落，显著提升了数据的可读性和决策价值。

4. 错误自愈能力：当 SQL 执行报错时（如首次使用了 SQLite 不支持的 RIGHT JOIN），Agent 能自动获取错误信息，LLM 分析原因后重新生成正确的 SQL，无需人工干预。Agent 配置了 `max_iterations=8`，即最多 8 轮自动修正。

## 七 开发过程问题记录与解决方案

在项目开发全过程中，共遇到并解决了 8 个技术问题，按发现来源分为"AI 协作中的自我测试发现"和"用户反馈"两类。这些问题的记录和解决过程体现了工程实践中的调试和迭代能力。

### 问题 1：`requirements.txt` 中包含 `python>=3.10` 导致 pip 安装失败

- 发现来源：用户反馈 `ERROR: No matching distribution found for python>=3.10`
- 原因分析：`python>=3.10` 是对运行环境的约束而非 pip 可安装的包名，pip 无法解析此条目
- 解决方案：从 `requirements.txt` 中删除 `python>=3.10` 行。Python 版本约束应在 README 中明确说明，而非放入 pip 依赖文件

### 问题 2：`agent` 目录缺少 `__init__.py` 导致模块导入失败

- 发现来源：AI 全流程测试时报 `ModuleNotFoundError: No module named 'agent.db_connection'`
- 原因分析：Python 需要目录中包含 `__init__.py`（即使是空文件）才能将其识别为包，否则无法使用 `from agent.xxx import ...` 语法
- 解决方案：在 `agent/` 目录下创建空的 `__init__.py` 文件，使其成为合法的 Python 包

### 问题 3：`NameError: name 'px' is not defined` — Streamlit 各 Tab 绘图报错

- 发现来源：用户反馈点击就业分析、考研分析、保研与留学 Tab 时均报错
- 原因分析：`import plotly.express as px` 被写在了第一个 Tab（数据概览）的代码块内部，而非文件顶部。Streamlit 每次切换 Tab 时会重新执行对应代码块，但不会执行未激活 Tab 的 import 语句。Tab 2/3/4 中使用了 `px` 但未在自己的作用域内导入
- 解决方案：将 `import plotly.express as px` 提升至文件顶部（第 14 行），与所有其他顶层 import 放在一起



#### 问题 4：AI 问答 Tab 推荐问题按钮点击无反应

- 发现来源：用户反馈
- 原因分析：推荐问题按钮仅设置了 `st.session_state.user_question = q`（将值绑定到 text_input），但用户仍需手动点击"提问"按钮，交互链路过长
- 解决方案：重构交互逻辑，推荐问题按钮点击后通过 `st.session_state.pending_question` 直接触发 Agent 查询，无需二次点击；手动输入改为 `st.form` 包裹
